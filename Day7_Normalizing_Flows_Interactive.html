📐 Mathematical Foundations

📦 Change of Variables Theorem

🔑
This theorem tells us: if you transform a random variable, the probability mass must be conserved. The area of the blue rectangle = area of the green rectangle (from the lecture figures).
Intuition (1D): Imagine squishing a blob of probability dough. If you stretch it in one direction, it gets thinner — the total mass (probability) stays the same.

Formal Theorem:

Let Z and X be random variables related by a mapping f : ℝⁿ → ℝⁿ such that X = f(Z) and Z = f⁻¹(X). Then:

p_X(x) = p_Z(f⁻¹(x)) · |det(∂f⁻¹(x)/∂x)| The density of X = density of Z at corresponding point × |det of Jacobian of inverse|

Equivalently, using z = f⁻¹(x):

p_X(x) = p_Z(z) · |det(∂f(z)/∂z)|⁻¹ Because for invertible A: det(A⁻¹) = [det(A)]⁻¹
Important constraints:
  • x and z must be continuous random variables of the same dimension.
  • The mapping f must be invertible.
  • ∂f⁻¹(x)/∂x is the n×n Jacobian matrix.

🔢 The Jacobian Matrix

🔍
The Jacobian describes how a transformation changes space locally. Think of it as a "local linear approximation" to a nonlinear transformation.

Formal Definition (2D Case):

Let f(x,y) = (u(x,y), v(x,y)) be a transformation from ℝ² → ℝ². The Jacobian matrix is:

J_f(x,y) =
∂u/∂x
∂u/∂y
∂v/∂x
∂v/∂y
Each entry (i,j) = how output dimension i changes w.r.t. input dimension j

Geometric Interpretation:

|det J| tells us how much area/volume is stretched or compressed:
  • |det J| > 1 → area expands
  • |det J| < 1 → area contracts
  • |det J| = 0 → local collapse (not invertible!)

This transformation scales area by a factor of 5!

⚡ Why Triangular Jacobians Are a Big Deal

Computing det of a general n×n matrix costs O(n³) — catastrophically slow for high-dimensional data (e.g., 256×256 images → n = 65,536!).

Problem: O(n³) = O(65536³) ≈ 2.8 × 10¹⁴ operations. That's not happening in a training loop.

But the determinant of a triangular matrix is just the product of its diagonal entries:

Key insight: Design transformations so their Jacobian is triangular → fast determinant → tractable training!

🌊 Normalizing Flow Models

🏗️ The Model Structure

In a normalizing flow model, the mapping between latent variables Z and observed variables X is deterministic and invertible:

G : ℝⁿ → ℝⁿ  |  X = G(Z)  |  Z = G⁻¹(X)

By the change of variables formula, the marginal likelihood p(x) is:

p_X(x; θ) = p_Z(G_θ⁻¹(x)) · |det(∂G_θ⁻¹(x)/∂x)| This is EXACT — no approximations, no lower bounds!

📖 Where the Name Comes From

🔄 "Normalizing"

The change of variables formula produces a normalized density after applying an invertible transformation. The total probability integrates to 1 — it's always a valid density.

⛓️ "Flow"

Invertible transformations can be composed (chained) to create more complex, expressive mappings. The "flow" refers to probability mass flowing through a sequence of transformations.

Composing flows: The determinant of the full Jacobian is the product of determinants of each individual transformation — so we just sum the log-determinants!

🎓 Learning and Inference

📚 Training (Maximum Likelihood):

max_θ log p(D; θ) = Σ_{x∈D} [ log π(G_θ⁻¹(x)) + log |det(∂G_θ⁻¹(x)/∂x)| ] We train G⁻¹, then use G for generation. Two terms: prior log-prob + log-Jacobian

🎲 Sampling (Generation):

z ~ π(z),    x = G_θ(z) Draw a random z from simple prior, pass through G

🔍 Inference (Latent Code):

z = G_θ⁻¹(x) No encoder needed! Just apply the inverse. Exact, no approximation.
No inference network needed! Unlike VAEs, we don't need a separate encoder. The inverse of G gives us z exactly.

⚙️ Requirements for a Good Flow

  1. Simple prior π(z) — e.g., standard Gaussian. Must support efficient sampling and tractable likelihood evaluation.
  2. Invertible transformations — We must be able to go both ways: X → Z and Z → X.
  3. Tractable Jacobian determinant — General det is O(n³). We need special structure.
The Big Design Question: How do we build invertible transformations that also have triangular Jacobians so we can compute determinants cheaply?

Answer: Coupling Layers! →

🔗 Coupling Layers

💡
Coupling layers are the secret weapon of normalizing flows. They're the building block that makes invertibility AND efficient Jacobian computation possible simultaneously.

The Coupling Layer Idea

Step 1 — Split the input:

Divide the input vector x into two parts: x₁ and x₂

x = [x₁, x₂]

Step 2 — Apply conditioned transformation:

y₁ = x₁                          ← Passthrough (unchanged)
y₂ = x₂ ⊙ exp(s(x₁)) + t(x₁)  ← Scale + Translate s = scale network, t = translation network, both take x₁ as input

✅ Why Coupling Layers Are Brilliant

🔄 Invertibility is trivial:

x₁ = y₁
x₂ = (y₂ - t(y₁)) ⊙ exp(-s(y₁)) Just rearrange! y₁ is always unchanged.

🔺 Triangular Jacobian:

diag(exp(s))
Upper-left: identity. Lower-right: diagonal. It's triangular!
Determinant of the Jacobian:
det J = ∏ exp(sᵢ(x₁)) = exp(∑ sᵢ(x₁))
This is O(n) — just a sum! And s and t can be arbitrarily complex neural networks.

✨ NICE: Non-linear Independent Components Estimation

📜 Overview

📅 Dinh et al., 2014 — The original normalizing flow architecture
Additive Coupling Volume-Preserving Rescaling Layer
✨

🔧 How NICE Works

Step 1 — Additive Coupling Layers:

NICE uses a simplified form of coupling — additive only (no scaling):

x_{1:d} = z_{1:d}                    ← Passthrough
x_{d+1:n} = z_{d+1:n} + H(z_{1:d}) ← Add neural network output H is an arbitrary neural network. Jacobian determinant = 1 (always!)
Why det J = 1? With only addition (no scaling), the Jacobian is a lower-triangular matrix with 1s on the diagonal. det = 1·1·...·1 = 1. Free lunch!

Step 2 — Compose Multiple Layers:

Stack several additive coupling layers with different partitions in each layer, so all dimensions eventually interact.

Step 3 — Final Rescaling Layer:

Since additive coupling is volume-preserving (det=1), NICE can't model distributions that require volume changes. A final diagonal scaling layer fixes this:

x = s ⊙ z s is a learned scaling vector. This introduces volume changes (non-unit det).

⚠️ NICE Limitation: The purely additive (volume-preserving) transformations limit expressiveness. The model cannot reshape the distribution in ways that require volume changes. RealNVP fixes this!

🔁 Inverting NICE

Because H is only added (not multiplied), the inverse is trivially easy:

z_{1:d} = x_{1:d}                         ← Same
z_{d+1:n} = x_{d+1:n} - H(x_{1:d})   ← Just subtract!
Key insight: H can be an arbitrarily complex (non-invertible!) neural network, yet the whole layer remains invertible. We only need H in the forward direction!

💜 RealNVP: Real-valued Non-Volume Preserving

📜 Overview

📅 Dinh et al., 2016 — Enhancement of NICE with scaling
Affine Coupling Non-Volume Preserving Alternating Partitions
💜

🆕 What Changes from NICE?

One key addition: RealNVP adds a scaling term to the coupling layer, making it an affine (not just additive) transformation. This breaks the volume-preserving constraint — in a good way!

RealNVP Coupling Layer:

x_{1:d} = z_{1:d}                               ← Passthrough
x_{d+1:n} = z_{d+1:n} ⊙ exp(F(z_{1:d})) + H(z_{1:d}) ← Scale AND translate F = scale network, H = translation network — both neural networks

NICE

y₂ = x₂ + t(x₁) Add only
det J = 1 always
→

RealNVP

y₂ = x₂ ⊙ exp(s(x₁)) + t(x₁) Scale + Add
det J = exp(∑s) ≠ 1

⚙️ Benefits of RealNVP

  • Efficient Jacobian det: Due to triangular structure → det = ∏ exp(sᵢ) = exp(∑sᵢ). Still O(n)!
  • Exact likelihood: Can directly compute p(x) with no approximations.
  • Efficient sampling: Forward pass through G(z) gives samples.
  • More expressive: Can model volume changes, capturing more complex distributions.
Alternating partitions: RealNVP alternates which half of the input is passed through. This ensures all dimensions are transformed across layers and the model is fully expressive.

🔁 Inverting RealNVP

z_{1:d} = x_{1:d}
z_{d+1:n} = (x_{d+1:n} - H(x_{1:d})) ⊙ exp(-F(x_{1:d})) Still easy! Just subtract t, then divide by exp(s) = multiply by exp(-s)
F and H never need to be inverted. They can be any neural network — arbitrarily complex — because we only ever evaluate them in the forward direction.

RealNVP produces noticeably sharper and more realistic image samples compared to NICE, thanks to the affine (scale + translate) coupling layers that allow volume changes.

✨ Glow: Generative Flow

📜 Overview

📅 Kingma & Dhariwal, OpenAI, 2018 — State-of-the-art generative flow
Invertible 1×1 Conv ActNorm Affine Coupling
✨

🆕 What's New in Glow?

Glow makes three major innovations over RealNVP:

1. ActNorm (Activation Normalization)

Applies a per-channel affine transformation (scale + bias) initialized using data statistics (mean and variance of the first mini-batch). This replaces Batch Normalization and works better for flow models.

y = (x - b) / s     where b, s initialized from data Invertible: x = y × s + b

2. Invertible 1×1 Convolutions

In RealNVP, the channel permutation (deciding which channels to pass/transform) is fixed. Glow replaces this with a learned invertible 1×1 convolution — essentially a learned permutation matrix generalized to a full invertible weight matrix.

y = W · x     where W is an invertible matrix (learned) det J = |det(W)|. Use LU decomposition for O(n) computation.
Why this matters: A fixed permutation only reorders channels. A learned invertible matrix can mix channels in any way, dramatically increasing expressiveness!

3. Affine Coupling Layers (as in RealNVP)

Same affine coupling layers as RealNVP — scale and translate conditioned on part of the input. Integrated with ActNorm and invertible 1×1 convolutions for greater flexibility.

ActNorm
Per-channel affine normalization
↓
Invertible 1×1 Convolution

↓

Scale + translate (from RealNVP)

This 3-step block is repeated K times per scale level, and L scale levels in total.

🏆 Key Takeaways (from lecture)

  1. Normalizing flows provide exact likelihoods — no variational lower bounds, no Monte Carlo estimates.
  2. Simple → Complex via invertible maps — transform a Gaussian into any distribution you want.
  3. Change of variables + Jacobian determinant — the mathematical core. Must be computed efficiently.
  4. Triangular Jacobians = O(n) det — the reason coupling layers are the go-to building block.
  5. Trade-off: expressiveness vs efficiency — more complex transformations = better models, but may be slower.

⚖️ Considerations

Expressiveness vs Efficiency: More expressive transformations (e.g., learned convolutions) are more powerful but harder to optimize. Simpler transformations (additive coupling) are fast but less flexible.
Jacobian must be tractable: The entire framework collapses if you can't compute (or approximate) the Jacobian determinant efficiently. Architectural choices must always consider this.
Dimensionality constraint: x and z must have the same dimension. This limits some applications compared to VAEs (which can have lower-dimensional z).
