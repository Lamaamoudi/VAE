① What Is ∂ ? (The "Curly d")

🌡️
One-line answer: ∂ means "how much does the output change when I nudge just ONE of the inputs, while keeping everything else frozen?"

Start with a regular derivative (one variable)

You know a car's speedometer shows speed — how fast your position changes over time. That's a derivative. If position = f(t) = t², then:

df/dt = 2t At t=3 seconds: speed = 2×3 = 6 m/s. The position changes 6 metres per second at that moment.

Now: what if there are TWO inputs?

Say temperature in a room depends on both position and time: T(x, t) = x² + 3t

Now you have two questions you can ask separately:

∂T/∂x — "How fast does temperature change as I walk across the room, at a frozen moment in time?"

∂T/∂x = 2x   (treat t as a constant — it disappears)
∂T/∂t — "How fast does temperature change as time passes, while standing still?"

∂T/∂t = 3   (treat x as a constant — x² disappears)

🔢 Step-by-step example: f(x, y) = x²y + 5y

QuestionRuleAnswerIntuition
∂f/∂x Treat y as a frozen number 2xy x² becomes 2x; the 5y term has no x so vanishes
∂f/∂y Treat x as a frozen number x² + 5 x²y → x²; 5y → 5; both just coefficients of y
Why does this appear in flow models?
When we write ∂x/∂z we're asking: "if I change the latent code z by a tiny amount, how much does the output x change?" That ratio tells us how the transformation is stretching or compressing space — which is exactly what we need to track probability.

② What Does "Invertible" Mean?

🔑
One-line answer: A function is invertible if you can always undo it — go backwards from output to input without any ambiguity.

The lock-and-key analogy

Think of a function f as a lock. Invertible means you have a key (f⁻¹) that always undoes it. For every locked state (output x), there is exactly one unlocked state (input z) it came from.

✅ Invertible example: f(z) = z + 5

Forward: z=3 → x = 3+5 = 8
Backward: x=8 → z = 8−5 = 3 ✓

f⁻¹(x) = x − 5   ← always works
❌ NOT invertible: f(z) = z²

Forward: z=3 → x = 9
Forward: z=−3 → x = 9 (same!)

Backward: x=9 → z = ??? (3 or −3?)
We can't tell — not invertible!

More examples — can you undo it?

FunctionInvertible?Inverse (if yes)Why / Why not
f(z) = 3z✅ Yesf⁻¹(x) = x/3Multiply by 3 → undo by dividing by 3
f(z) = z + c✅ Yesf⁻¹(x) = x − cShift right → undo by shifting left
f(z) = |z|❌ No—f(2)=2 and f(−2)=2. Two inputs → same output
f(z) = sin(z)❌ No (on all ℝ)—Repeats every 2π — infinitely many inputs → same output
f(z) = ez✅ Yesf⁻¹(x) = ln(x)Exponential → undo with logarithm

Why does this matter for flows?

In a normalizing flow, we transform z (simple Gaussian) → x (complex data). We MUST be able to go backwards too (x → z), because:
  • Training: we observe x, need to compute z = f⁻¹(x) to evaluate the likelihood
  • Generation: we sample z, then compute x = f(z)
If f is not invertible, we're stuck — we can't get z from x.

③ What Is a Determinant — and Why Does It Exist?

📐
One-line answer: The determinant of a matrix is a single number that tells you how much the matrix stretches, compresses, or flips space. It's the "volume scaling factor."

Forget matrices for a second — think about scaling

Imagine you have a square with area = 1. You apply a transformation. What's the area afterwards? The determinant answers that question.

④ Why Is |det| in the Probability Formula?

🎯
Core reason: Probability mass is like water — it can't appear or disappear. When a transformation stretches a region, the probability density over that region must thin out proportionally to compensate.

🚰 The Water Pipe Analogy

Imagine water flowing through a pipe. If the pipe gets twice as wide, the water flows half as fast (for the same total flow). Probability works the same way:

🟦
Z region
width = 1
density = p_Z = 0.5
→ ×2 stretch →
🟩🟩
X region
width = 2
density = p_X = 0.25
The region doubled in size → density halved. The factor is 1/|det J| = 1/2.
Total probability = density × width: 0.5×1 = 0.5×1 and 0.25×2 = 0.5×1 ✅ same!

🔢 Seeing the formula piece by piece

The full formula is:

p_X(x)  =  p_Z( f⁻¹(x) )  ×  |det J|⁻¹ = "density at the corresponding z point" × "correction for how much space was stretched"
PieceWhat it means in plain English
p_X(x) The probability density of the output x — this is what we want to know
p_Z(f⁻¹(x)) Find the input z that maps to x (by applying the inverse f⁻¹), then evaluate the simple known density at that z
|det J|⁻¹ The correction factor: divide by how much the transformation stretched things at that point. If it stretched by 3×, density is 3× thinner → multiply by 1/3
Why ⁻¹? Because stretching (big det) → density goes DOWN. Compressing (small det) → density goes UP. They're inverses of each other. When det=1 (no stretching), |det|⁻¹ = 1 and the formula reduces to p_X(x) = p_Z(z). Makes sense!

⑤ Why Does ∂f⁻¹(x)/∂x Sometimes Appear as ∂z/∂x — and Is That the Same Thing?

🔁
Short answer: Yes, they're the same. Since z = f⁻¹(x), writing "∂z/∂x" is just a shorthand for "∂f⁻¹(x)/∂x".

How substitution works

We have two variables connected by a function: x = f(z), which means z = f⁻¹(x).

When you see "∂f⁻¹(x)/∂x" in a formula, you can rename f⁻¹(x) as just "z" because that's all it is — the variable z expressed as a function of x. So:

∂f⁻¹(x)/∂x  ≡  ∂z/∂x These are identical. The second is just cleaner notation.

🔢 Concrete example to make it stick

Let f(z) = 2z + 1, so x = 2z + 1. The inverse is z = f⁻¹(x) = (x−1)/2.

Way of writing itComputationResult
∂f⁻¹(x)/∂x Derivative of (x−1)/2 with respect to x 1/2
∂z/∂x Since z = (x−1)/2, derivative of z w.r.t. x 1/2
(∂f/∂z)⁻¹  (third way!) f'(z) = 2, so 1/f'(z) = 1/2 1/2
All three notations give the same number. The lecture uses different ones in different places — now you know they're just different ways to write the same thing.

Why does the variable name flip between x and z?

The change of variables formula has two equivalent forms depending on which direction you're thinking about:

"I have x, I want p_X(x)"
→ Use the inverse: z = f⁻¹(x)
→ Jacobian: ∂z/∂x (how z changes with x)
→ Formula: p_Z(z) · |∂z/∂x|
"I have z, I want p_X(x)"
→ Forward: x = f(z)
→ Jacobian: ∂x/∂z (how x changes with z)
→ Formula: p_Z(z) · |∂x/∂z|⁻¹
Key: (∂z/∂x) and (∂x/∂z)⁻¹ are the same number — one is just the reciprocal of the other! The lecture sometimes uses one form, sometimes the other. Don't let it confuse you — they're equivalent.

📐 Mathematical Foundations

📦 Change of Variables Theorem

🔑
This theorem tells us: if you transform a random variable, the probability mass must be conserved. The area of the blue rectangle = area of the green rectangle (from the lecture figures).
Intuition (1D): Imagine squishing a blob of probability dough. If you stretch it in one direction, it gets thinner — the total mass (probability) stays the same.

Formal Theorem:

Let Z and X be random variables related by a mapping f : ℝⁿ → ℝⁿ such that X = f(Z) and Z = f⁻¹(X). Then:

\[ p_X(x) = p_Z\!\left(f^{-1}(x)\right) \cdot \left|\det\!\left(\frac{\partial f^{-1}(x)}{\partial x}\right)\right| \] The density of X = density of Z at corresponding point × |det of Jacobian of inverse|

Equivalently, using z = f⁻¹(x):

\[ p_X(x) = p_Z(z) \cdot \left|\det\!\left(\frac{\partial f(z)}{\partial z}\right)\right|^{-1} \] Because for invertible A: det(A⁻¹) = [det(A)]⁻¹
Important constraints:
  • x and z must be continuous random variables of the same dimension.
  • The mapping f must be invertible.
  • ∂f⁻¹(x)/∂x is the n×n Jacobian matrix.

🔢 The Jacobian Matrix

🔍
The Jacobian describes how a transformation changes space locally. Think of it as a "local linear approximation" to a nonlinear transformation.

Formal Definition (2D Case):

Let f(x,y) = (u(x,y), v(x,y)) be a transformation from ℝ² → ℝ². The Jacobian matrix is:

\[ J_f(x,y) = \begin{pmatrix} \dfrac{\partial u}{\partial x} & \dfrac{\partial u}{\partial y} \\[10pt] \dfrac{\partial v}{\partial x} & \dfrac{\partial v}{\partial y} \end{pmatrix} \] Each entry (i,j) = how output dimension i changes w.r.t. input dimension j

Geometric Interpretation:

|det J| tells us how much area/volume is stretched or compressed:
  • |det J| > 1 → area expands
  • |det J| < 1 → area contracts
  • |det J| = 0 → local collapse (not invertible!)

🎯 Worked Example (from Appendix):

Let f(x,y) = (2x + y, x + 3y). Then:

\[ J_f = \begin{pmatrix} 2 & 1 \\ 1 & 3 \end{pmatrix}, \qquad \det J = 2 \cdot 3 - 1 \cdot 1 = \color{#e94560}{5} \] This transformation scales area by a factor of 5!

⚡ Why Triangular Jacobians Are a Big Deal

Computing det of a general n×n matrix costs O(n³) — catastrophically slow for high-dimensional data (e.g., 256×256 images → n = 65,536!).

Problem: O(n³) = O(65536³) ≈ 2.8 × 10¹⁴ operations. That's not happening in a training loop.

But the determinant of a triangular matrix is just the product of its diagonal entries:

\[ \det\begin{pmatrix} a_{11} & 0 & 0 \\ * & a_{22} & 0 \\ * & * & a_{33} \end{pmatrix} = a_{11} \times a_{22} \times a_{33} \] O(n) instead of O(n³) — this is the key design goal!
Key insight: Design transformations so their Jacobian is triangular → fast determinant → tractable training!

🍕 Change of Variables — Explained With Pizza

🤔
Why do we need this? We know the density of Z (simple Gaussian). We want to know the density of X = f(Z). The change of variables formula is the bridge.

🍕 Analogy: Stretching Pizza Dough

Imagine you have a round ball of pizza dough (your Gaussian distribution). You press and stretch it into a flat shape (a complex distribution). The total amount of dough never changes — it just gets redistributed. If you stretched one area thin, the density there is lower. If you compressed another area thick, the density is higher.

The Jacobian determinant = how much you stretched or squished the dough at each point. If |det J| = 3, that region got stretched to 3× its original area — so the density is 3× thinner there.

📏 Concrete 1D Example — Step by Step

Let Z ~ Uniform(0, 1) so p_Z(z) = 1 for z ∈ [0,1]. Now let X = f(Z) = 2Z.

StepWhat we doResult
1Find the inverse: z = f⁻¹(x) = x/2z = x/2
2Compute the Jacobian (just df⁻¹/dx in 1D)df⁻¹/dx = 1/2
3Apply the formula: p_X(x) = p_Z(f⁻¹(x)) · |df⁻¹/dx|p_X(x) = 1 · |1/2| = 1/2
4Check: X ranges over [0, 2], and 1/2 × 2 = 1 ✅Still a valid density!
Intuition: f(Z)=2Z doubles the range from [0,1] to [0,2]. To keep total probability = 1, the density must be halved from 1 to 1/2. The Jacobian factor |1/2| does exactly that automatically.

📐 Concrete 2D Example — Step by Step

Let Z = (z₁, z₂) with p_Z(z) = 1 (uniform on unit square). Apply f(z) = (2z₁, 3z₂).

StepComputation

🧭 The Jacobian — A GPS for Transformations

🗺️
Think of the Jacobian as a map that tells you: "if I move a tiny bit in the z direction, how much does x move, and in which direction?" It captures ALL the local stretching, rotating, and shearing of the transformation.

🏗️ Building the Jacobian — Recipe

For a function f : ℝⁿ → ℝⁿ with outputs (x₁, x₂, ..., xₙ) and inputs (z₁, z₂, ..., zₙ):

Row \(i\) of \(J\) = "how does output \(x_i\) change w.r.t. each input?"
Column \(j\) of \(J\) = "how does each output change w.r.t. input \(z_j\)?"

\[ J[i,j] = \frac{\partial x_i}{\partial z_j} \] Entry at row i, column j = partial derivative of i-th output w.r.t. j-th input

🔢 Full Worked Example: f(z₁,z₂) = (z₁², z₁·z₂)

EntryFormula= ?
J[1,1] = ∂x₁/∂z₁∂(z₁²)/∂z₁2z₁
J[1,2] = ∂x₁/∂z₂∂(z₁²)/∂z₂0 (no z₂ in x₁)
J[2,1] = ∂x₂/∂z₁∂(z₁z₂)/∂z₁z₂
J[2,2] = ∂x₂/∂z₂∂(z₁z₂)/∂z₂z₁
\[ J = \begin{pmatrix} 2z_1 & 0 \\ z_2 & z_1 \end{pmatrix}, \qquad \det J = 2z_1 \cdot z_1 - 0 \cdot z_2 = \color{#e94560}{2z_1^2} \] At point z=(2,3): det J = 2×4 = 8 → this transformation expands area by 8× at that point

🔺 Why Triangular Jacobians Are the Holy Grail

When J is triangular (all zeros above or below the diagonal), computing its determinant is trivially just multiplying the diagonal:

General 3×3 det:
Need 6 multiplications + additions. For n×n: O(n³). At n=1000 that's 10⁹ operations per forward pass. 💀
Triangular 3×3 det:
det = a₁₁ × a₂₂ × a₃₃ — just 2 multiplications! For n×n: O(n). At n=1000 that's 1000 operations. ✅

Coupling layers are specifically designed so their Jacobian is always triangular. That's not a coincidence — it's the whole design philosophy.

🌊 Normalizing Flow Models

🏗️ The Model Structure

In a normalizing flow model, the mapping between latent variables Z and observed variables X is deterministic and invertible:

\[ G : \mathbb{R}^n \to \mathbb{R}^n \;|\; X = G(Z) \;|\; Z = G^{-1}(X) \]

By the change of variables formula, the marginal likelihood p(x) is:

\[ p_X(x;\,\theta) = p_Z\!\left(G_\theta^{-1}(x)\right) \cdot \left|\det\!\left(\frac{\partial G_\theta^{-1}(x)}{\partial x}\right)\right| \] This is EXACT — no approximations, no lower bounds!

📖 Where the Name Comes From

🔄 "Normalizing"

The change of variables formula produces a normalized density after applying an invertible transformation. The total probability integrates to 1 — it's always a valid density.

⛓️ "Flow"

Invertible transformations can be composed (chained) to create more complex, expressive mappings. The "flow" refers to probability mass flowing through a sequence of transformations.

Composing flows: The determinant of the full Jacobian is the product of determinants of each individual transformation — so we just sum the log-determinants!

🎓 Learning and Inference

📚 Training (Maximum Likelihood):

\[ \max_\theta \log p(D;\,\theta) = \sum_{x \in D} \left[ \log \pi\!\left(G_\theta^{-1}(x)\right) + \log\left|\det\!\left(\frac{\partial G_\theta^{-1}(x)}{\partial x}\right)\right| \right] \] We train G⁻¹, then use G for generation. Two terms: prior log-prob + log-Jacobian

🎲 Sampling (Generation):

\[ z \sim \pi(z), \qquad x = G_\theta(z) \] Draw a random z from simple prior, pass through G

🔍 Inference (Latent Code):

\[ z = G_\theta^{-1}(x) \] No encoder needed! Just apply the inverse. Exact, no approximation.
No inference network needed! Unlike VAEs, we don't need a separate encoder. The inverse of G gives us z exactly.

⚙️ Requirements for a Good Flow

  1. Simple prior π(z) — e.g., standard Gaussian. Must support efficient sampling and tractable likelihood evaluation.
  2. Invertible transformations — We must be able to go both ways: X → Z and Z → X.
  3. Tractable Jacobian determinant — General det is O(n³). We need special structure.
The Big Design Question: How do we build invertible transformations that also have triangular Jacobians so we can compute determinants cheaply?

Answer: Coupling Layers! →

🔗 Coupling Layers

💡
Coupling layers are the secret weapon of normalizing flows. They're the building block that makes invertibility AND efficient Jacobian computation possible simultaneously.

The Coupling Layer Idea

Step 1 — Split the input:

Divide the input vector x into two parts: x₁ and x₂

\[ x = [x_1,\; x_2] \]

Step 2 — Apply conditioned transformation:

\( y_1 = x_1 \)                          ← Passthrough (unchanged)
\( y_2 = x_2 \odot \exp(s(x_1)) + t(x_1) \)  ← Scale + Translate s = scale network, t = translation network, both take x₁ as input

✅ Why Coupling Layers Are Brilliant

🔄 Invertibility is trivial:

\( x_1 = y_1 \)
\( x_2 = (y_2 - t(y_1)) \odot \exp(-s(y_1)) \) Just rearrange! y₁ is always unchanged.

🔺 Triangular Jacobian:

\[ J = \begin{pmatrix} I & 0 \\ * & \mathrm{diag}(\exp(s)) \end{pmatrix} \] Upper-left: identity. Lower-right: diagonal. It's triangular!
Determinant of the Jacobian:
\(\det J = \prod_i \exp(s_i(x_1)) = \exp\!\left(\sum_i s_i(x_1)\right)\)
This is O(n) — just a sum! And s and t can be arbitrarily complex neural networks.

This is why the training loss uses \(\log|\det J| = \sum_i s_i(z_1)\) — just a sum of the scale network outputs!

✨ NICE: Non-linear Independent Components Estimation

📜 Overview

📅 Dinh et al., 2014 — The original normalizing flow architecture
Additive Coupling Volume-Preserving Rescaling Layer
✨

🔧 How NICE Works

Step 1 — Additive Coupling Layers:

NICE uses a simplified form of coupling — additive only (no scaling):

\( x_{1:d} = z_{1:d} \)                    ← Passthrough
\( x_{d+1:n} = z_{d+1:n} + H(z_{1:d}) \) ← Add neural network output H is an arbitrary neural network. Jacobian determinant = 1 (always!)
Why det J = 1? With only addition (no scaling), the Jacobian is a lower-triangular matrix with 1s on the diagonal. det = 1·1·...·1 = 1. Free lunch!

Step 2 — Compose Multiple Layers:

Stack several additive coupling layers with different partitions in each layer, so all dimensions eventually interact.

Step 3 — Final Rescaling Layer:

Since additive coupling is volume-preserving (det=1), NICE can't model distributions that require volume changes. A final diagonal scaling layer fixes this:

\[ x = s \odot z \] s is a learned scaling vector. This introduces volume changes (non-unit det).

⚠️ NICE Limitation: The purely additive (volume-preserving) transformations limit expressiveness. The model cannot reshape the distribution in ways that require volume changes. RealNVP fixes this!

🔁 Inverting NICE

Because H is only added (not multiplied), the inverse is trivially easy:

\( z_{1:d} = x_{1:d} \)                         ← Same
\( z_{d+1:n} = x_{d+1:n} - H(x_{1:d}) \)   ← Just subtract!
Key insight: H can be an arbitrarily complex (non-invertible!) neural network, yet the whole layer remains invertible. We only need H in the forward direction!

Why det J = 1? The Jacobian diagonal entries are all 1 (from the identity for x₁, and coefficient 1 for x₂ = z₂ + constant). So det = 1×1×1×1 = 1. The transformation only shifts things, never stretches.

📉 The Volume-Preserving Limitation — Illustrated

Imagine you want to model a distribution that looks like a squished pancake — concentrated along one axis, spread on another. To do this you need to scale the axes — multiply by different amounts. But NICE can only add offsets, never scale. It's like trying to reshape a clay ball using only sliding motions but never squeezing.

NICE can do:
✅ Shift regions of probability mass
✅ Capture complex non-linear dependencies
❌ Change the "spread" of dimensions
❌ Model distributions with different scales per dimension
The rescaling fix:
NICE adds a final layer: x = s ⊙ z
where s is a learned per-dimension scale.
Now det J = ∏ sᵢ — but it's just one fixed layer, not per-sample adaptive scaling.
Bottom line: NICE proved the concept works beautifully. Its images look reasonable. But RealNVP (2 years later) removed the volume-preserving constraint by adding per-sample adaptive scaling — and got dramatically more expressive.

💜 RealNVP: Real-valued Non-Volume Preserving

📜 Overview

📅 Dinh et al., 2016 — Enhancement of NICE with scaling
Affine Coupling Non-Volume Preserving Alternating Partitions
💜

🆕 What Changes from NICE?

One key addition: RealNVP adds a scaling term to the coupling layer, making it an affine (not just additive) transformation. This breaks the volume-preserving constraint — in a good way!

RealNVP Coupling Layer:

\( x_{1:d} = z_{1:d} \)                               ← Passthrough
\( x_{d+1:n} = z_{d+1:n} \odot \exp(F(z_{1:d})) + H(z_{1:d}) \) ← Scale AND translate F = scale network, H = translation network — both neural networks

NICE

\[ y_2 = x_2 + t(x_1) \] Add only
\(\det J = 1\) always
→

RealNVP

\[ y_2 = x_2 \odot \exp(s(x_1)) + t(x_1) \] Scale + Add
\(\det J = \exp(\sum s) \neq 1\)

⚙️ Benefits of RealNVP

  • Efficient Jacobian det: Due to triangular structure → \(\det = \prod \exp(s_i) = \exp(\sum s_i)\). Still O(n)!
  • Exact likelihood: Can directly compute p(x) with no approximations.
  • Efficient sampling: Forward pass through G(z) gives samples.
  • More expressive: Can model volume changes, capturing more complex distributions.
Alternating partitions: RealNVP alternates which half of the input is passed through. This ensures all dimensions are transformed across layers and the model is fully expressive.

🔁 Inverting RealNVP

\( z_{1:d} = x_{1:d} \)
\( z_{d+1:n} = \bigl(x_{d+1:n} - H(x_{1:d})\bigr) \odot \exp\bigl(-F(x_{1:d})\bigr) \) Still easy! Just subtract t, then divide by exp(s) = multiply by exp(-s)
F and H never need to be inverted. They can be any neural network — arbitrarily complex — because we only ever evaluate them in the forward direction.

The exp(F(z₁)) factor is what makes RealNVP "Non-Volume Preserving". exp(0.4)≈1.49 stretches z₂[0], while exp(−0.2)≈0.82 compresses z₂[1]. The distribution's shape changes in a way NICE simply cannot do.

🎨 Why Does This Matter for Image Generation?

Real images have pixels with very different statistical properties. Some regions (sky, uniform background) have low variance — the distribution should be concentrated (compressed). Other regions (edges, textures) have high variance — the distribution should be spread out (stretched). RealNVP's per-sample adaptive scaling lets it capture these different variances. NICE can't, because its only tool is translation.

⚙️ The Training Signal

During training, the loss for one sample x is:

\[ \mathcal{L} = -\log p(x) = -\log \pi(G^{-1}(x)) - \log|\det J| = \tfrac{1}{2}\|z\|^2 + \mathrm{const} - \sum_i F_i(z_1) \] The model pushes z toward zero (Gaussian) while maximizing log|det J| = Σ Fᵢ to avoid trivial solutions

✨ Glow: Generative Flow

📜 Overview

📅 Kingma & Dhariwal, OpenAI, 2018 — State-of-the-art generative flow
Invertible 1×1 Conv ActNorm Affine Coupling
✨

🆕 What's New in Glow?

Glow makes three major innovations over RealNVP:

1. ActNorm (Activation Normalization)

Applies a per-channel affine transformation (scale + bias) initialized using data statistics (mean and variance of the first mini-batch). This replaces Batch Normalization and works better for flow models.

\[ y = \frac{x - b}{s} \quad \text{where } b, s \text{ initialized from data} \] Invertible: x = y × s + b

2. Invertible 1×1 Convolutions

In RealNVP, the channel permutation (deciding which channels to pass/transform) is fixed. Glow replaces this with a learned invertible 1×1 convolution — essentially a learned permutation matrix generalized to a full invertible weight matrix.

\[ y = W \cdot x \quad \text{where } W \text{ is an invertible matrix (learned)} \] det J = |det(W)|. Use LU decomposition for O(n) computation.
Why this matters: A fixed permutation only reorders channels. A learned invertible matrix can mix channels in any way, dramatically increasing expressiveness!

3. Affine Coupling Layers (as in RealNVP)

Same affine coupling layers as RealNVP — scale and translate conditioned on part of the input. Integrated with ActNorm and invertible 1×1 convolutions for greater flexibility.

🔄 Innovation 2: Invertible 1×1 Convolution — What Is It Really?

In RealNVP, at each coupling layer some channels are passed through and others are transformed. The split is fixed — e.g., always the first half vs second half. This is limiting because the first half never gets transformed by its own network.

RealNVP's fixed channel split: [ch1, ch2, ch3, ch4] → always split at ch2|ch3. The coupling always sees the same "context" channels.

An invertible 1×1 convolution is really just a learned W matrix applied to the channel dimension. Think of it as: before each coupling layer, remix/permute the channels in a learned way.

Input: \([c_1, c_2, c_3, c_4]\) — 4 channels

\(W\) = random invertible \(4{\times}4\) matrix (learned)

Output: \([c_1', c_2', c_3', c_4']^T = W \cdot [c_1, c_2, c_3, c_4]^T\)

\(\log|\det J| = H \times W_{\mathrm{sp}} \times \log|\det(W)|\)  ← just the log-det of the weight matrix! H×W = spatial dimensions of feature map. Computed efficiently via LU decomposition of W.
What LU decomposition does: Any matrix W = L × U (lower-triangular × upper-triangular). Then det(W) = det(L) × det(U) = product of diagonal of L × product of diagonal of U. This is O(n) once you have L and U!

🏆 Key Takeaways (from lecture)

  1. Normalizing flows provide exact likelihoods — no variational lower bounds, no Monte Carlo estimates.
  2. Simple → Complex via invertible maps — transform a Gaussian into any distribution you want.
  3. Change of variables + Jacobian determinant — the mathematical core. Must be computed efficiently.
  4. Triangular Jacobians = O(n) det — the reason coupling layers are the go-to building block.
  5. Trade-off: expressiveness vs efficiency — more complex transformations = better models, but may be slower.

⚖️ Considerations

Expressiveness vs Efficiency: More expressive transformations (e.g., learned convolutions) are more powerful but harder to optimize. Simpler transformations (additive coupling) are fast but less flexible.
Jacobian must be tractable: The entire framework collapses if you can't compute (or approximate) the Jacobian determinant efficiently. Architectural choices must always consider this.
Dimensionality constraint: x and z must have the same dimension. This limits some applications compared to VAEs (which can have lower-dimensional z).
