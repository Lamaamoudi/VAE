Step 0 — The Setup
What Are We Actually Trying to Do?
Before any math, understand the goal
🌍 Real-World Analogy: The Nature Photographer

Imagine you're a nature photographer trying to understand "the distribution of animals in a forest". You can't observe every animal at every moment — you only have a finite number of photos (your training data). Your goal is to build a mental model (M̂) that so accurately captures how animals appear in that forest that if someone asked "what's the probability of seeing a fox at dawn?", your model would give a good answer.

In our case: forest → data domain, photos → training examples, mental model → Pθ, true animal distribution → Pdata.

1

There exists a true data distribution Pdata

This is the "ground truth" — the real probability distribution that generated all the data you see. You never get to observe it directly. For images, it's the probability of every possible image existing in the world.

2

You have a model Pθ with learnable parameters θ

This is your approximation — a neural network or formula whose output is a probability distribution. By adjusting θ, you change the shape of Pθ to hopefully match Pdata.

3

You want Pθ ≈ Pdata

But you only have a finite dataset D sampled from Pdata. You can't see the full distribution. So the question becomes: how do you measure "closeness" between two distributions? That's where KL divergence comes in.

⚠️
Why can't we just match Pθ = Pdata exactly?
For MNIST: 784 binary pixels → 2784 ≈ 10236 possible images. Even with 107 training examples, each one is a single point in a space of 10236 — like trying to understand the ocean by sampling 10 million drops. We need a model that generalizes.
Step 1 — Measuring Distance
KL Divergence
How do you measure how "wrong" your model is?
📡 Analogy: The Compression / Messaging Cost

Imagine you're sending messages over a channel. The optimal encoding for a distribution P uses exactly -log P(x) bits per message. If you design your code based on Pθ (your model), but the actual messages come from Pdata (reality), you'll use more bits than necessary. KL divergence = the extra bits you waste per message because you used the wrong distribution to design your code.

KL Divergence Formula
\[ D_{\text{KL}}(P_{\text{data}} \| P_\theta) = \mathbb{E}_{x \sim P_{\text{data}}} \!\left[\log \frac{P_{\text{data}}(x)}{P_\theta(x)}\right] = \sum_x P_{\text{data}}(x) \cdot \log \frac{P_{\text{data}}(x)}{P_\theta(x)} \]

Reading the Formula Piece by Piece

1

We sample x ~ Pdata

We're taking the expectation under the real distribution — we care about errors on data that actually occurs in the world.

2

We compute log(Pdata(x) / Pθ(x))

This is the log-ratio of the real probability to what our model thinks. If our model is correct, Pθ(x) = Pdata(x), so the ratio = 1 and log(1) = 0 — no waste!

3

We average over all x

Weighted by how often each x actually occurs (Pdata(x)). Rare events count less; common events count more.

Three Critical Properties

✅
Property 1: DKL ≥ 0 always
KL divergence is always non-negative. It equals exactly 0 if and only if the two distributions are identical everywhere. This is proven by Jensen's inequality: since log is concave, E[log(P/Q)] ≤ log(E[P/Q]) = log(1) = 0... wait, that's for -KL. KL itself uses log(P/Q) and P≥0 so the negative cannot be constructed without Q dominating P.
🔄
Property 2: NOT symmetric — DKL(P‖Q) ≠ DKL(Q‖P)
Example: If Pdata says "cats definitely exist" (P=0.9) but your model says "cats are impossible" (Pθ=0.001), you get a HUGE penalty from D(Pdata‖Pθ). But D(Pθ‖Pdata) would weight this event by Pθ ≈ 0, nearly ignoring it. This asymmetry matters!
💡
Property 3: Zero probability = infinite penalty
If Pθ(x) = 0 for any x that actually occurs (Pdata(x) > 0), then log(Pdata/0) = log(∞) = ∞ — the KL divergence blows up to infinity! This is why models must assign non-zero probability to everything in the support of Pdata. The log penalty enforces this.

Step 2 — What Your Model Needs to Learn
Log-Likelihood
Why do we use log? What does it actually mean?
🎰 Analogy: The Prediction Game

Every time a training example x appears, your model must "bet" on it. Your score is log Pθ(x). If Pθ(x) = 1.0 (perfect confidence, correct), you score 0 — best possible. If Pθ(x) = 0.5 (uncertain), you score log(0.5) = -0.69 — moderate penalty. If Pθ(x) = 0.001 (very wrong), you score log(0.001) = -6.9 — big penalty. If Pθ(x) = 0 (impossible according to model), you score -∞ — game over!

Why log? Three reasons:

1️⃣
Products become sums (numerical stability)
The joint probability of n independent events is P(x₁)·P(x₂)·…·P(xₙ). With n=784 values each between 0 and 1, this product underflows to zero on any computer. Taking the log turns it into a sum: log(P(x₁)) + log(P(x₂)) + … which is perfectly computable.
2️⃣
Heavy penalty for near-zero probabilities
The log function heavily penalizes small probabilities. This forces the model to assign non-zero probability to all data it might encounter. If Pθ(x) → 0 for any real data point, log Pθ(x) → -∞ — the model gets an infinite loss signal.
3️⃣
Natural connection to information theory
-log₂ P(x) is exactly the number of bits needed to encode x under distribution P. Maximizing log-likelihood = minimizing the encoding length of your data = finding the most efficient compression.
Expected Log-Likelihood (what we want to maximize)
\[ \mathcal{L}(\theta) = \mathbb{E}_{x \sim P_{\text{data}}}[\log P_\theta(x)] = \sum_x P_{\text{data}}(x) \cdot \log P_\theta(x) \]

This asks: "On average, how much log-probability does my model assign to real data?" Maximize this = make your model think real data is likely.

Step 3 — The Magic Bridge
The Key Algebraic Trick
Why minimizing KL divergence = maximizing log-likelihood

This is the most important step. We want to minimize DKL(Pdata‖Pθ) — but that's hard to compute since we don't know Pdata. Here's how we bypass that:

Step-by-step derivation
\[ D_{\text{KL}}(P_{\text{data}} \| P_\theta) = \mathbb{E}_{x \sim P_{\text{data}}}\!\left[\log \frac{P_{\text{data}}(x)}{P_\theta(x)}\right] \] \[ = \mathbb{E}_{x \sim P_{\text{data}}}[\log P_{\text{data}}(x)] \;-\; \mathbb{E}_{x \sim P_{\text{data}}}[\log P_\theta(x)] \] \[ = \underbrace{-H(P_{\text{data}})}_{\text{constant (doesn't depend on } \theta)} \;-\; \underbrace{\mathbb{E}_{x \sim P_{\text{data}}}[\log P_\theta(x)]}_{\text{expected log-likelihood}} \]

🔑
The key insight: −H(Pdata) is a constant with respect to θ. It doesn't change no matter what parameters we choose! So when we ask "what θ minimizes KL?", the constant term is irrelevant. We only need to maximize the part that depends on θ:
minθ DKL  =  maxθ 𝔼[log Pθ(x)]

Why is −H(Pdata) constant?

H(Pdata) is the entropy of the real data distribution — a fixed property of the world, not of your model. Think of it as "how complex/random is real data?" — that's a fact about the universe, not something you can change by tweaking θ. You're stuck with it.

⚠️
Consequence of this: We never know the true value of DKL! We can measure 𝔼[log Pθ(x)], but since we don't know H(Pdata), we can't compute DKL itself. We only know we're minimizing it. This is why we can't say "our model achieves KL = 0.3" — we can only compare models relative to each other.
Step 4 — Making it Practical
Maximum Likelihood Estimation (MLE)
We can't compute 𝔼[log Pθ(x)] exactly — so we approximate it
📊 Analogy: Polling vs. Census

To know 𝔼x ~ Pdata[log Pθ(x)], you'd need to average over every possible x weighted by Pdata(x) — this requires knowing Pdata exactly, which you don't have. It's like wanting to know the average income in a country by interviewing every single citizen (impossible). Instead, you poll a sample (your dataset D) and take the sample average — this is MLE!

From expectation → sample average → MLE
\[ \underbrace{\mathbb{E}_{x \sim P_{\text{data}}}[\log P_\theta(x)]}_{\text{true expected log-likelihood}} \;\approx\; \underbrace{\frac{1}{|D|}\sum_{x \in D} \log P_\theta(x)}_{\text{empirical log-likelihood (what we actually compute)}} \]

By the Law of Large Numbers: as |D| → ∞, the sample average converges to the true expectation.

MLE Objective (what you optimize in practice)
\[ \hat{\theta}_{\text{MLE}} = \arg\max_{\theta} \;\frac{1}{|D|}\sum_{x \in D} \log P_\theta(x) \]

In practice you minimize the negative log-likelihood (NLL): minθ −Σ log Pθ(x). Gradient descent minimizes, so flip the sign.

The Full Chain of Logic (summary)

1

Goal: Pθ ≈ Pdata

We want our model to match the true data distribution.

2

Measure closeness with KL divergence

DKL(Pdata‖Pθ) = 0 means perfect. The smaller the better.

3

Algebraic trick: KL = constant − 𝔼[log Pθ]

Since the constant doesn't depend on θ, minimizing KL ≡ maximizing expected log-likelihood.

4

Approximate expectation with dataset average

Replace 𝔼[log Pθ] with (1/|D|)Σ log Pθ(x). Now it's computable!

5

Result: MLE — maximize log-likelihood over training data

This is exactly what neural network training does when you minimize cross-entropy loss. They're the same thing!

💡
Cross-entropy loss = negative log-likelihood!
When you train a classifier with cross-entropy loss, you're literally doing MLE. You're maximizing the log-probability your model assigns to the correct class label. For autoregressive models: each step predicts the next token → the cross-entropy loss at each step is exactly -log Pθ(xi | x<i).

Autoregressive: Exact Likelihood

AR Objective (exact)
\[ \log p_\theta(x) = \sum_{i=1}^{n} \log p_\theta(x_i \mid x_{Every term is directly computable. No approximation. Training = maximize this sum (MLE).

VAE: Evidence Lower BOund (ELBO)

⚠️
The core problem with VAEs: The true likelihood involves integrating over all possible latent codes z:
pθ(x) = ∫ pθ(x|z) p(z) dz
This integral is intractable — you can't compute it for high-dimensional z. So VAEs optimize a lower bound instead.
VAE Objective: ELBO (lower bound on log-likelihood)
\[ \log p_\theta(x) \;\geq\; \underbrace{\mathbb{E}_{z \sim q_\phi(z|x)}\!\big[\log p_\theta(x \mid z)\big]}_{\text{reconstruction term}} \;-\; \underbrace{D_{\text{KL}}\big(q_\phi(z|x) \;\|\; p(z)\big)}_{\text{regularization term}} \]
🔁
Reconstruction term: 𝔼[log pθ(x|z)]

"Given a sampled latent code z, how well does the decoder reconstruct the original x?" This is like the standard log-likelihood, but conditioned on z. Maximizing this makes the decoder output look like the input.

🎯
Regularization term: DKL(qφ(z|x) ‖ p(z))

"How far is the encoder's output distribution from the prior p(z) = N(0,I)?" Minimizing this KL keeps the latent space organized and prevents it from collapsing. It forces the latent codes to form a nice, smooth Gaussian space you can sample from.

💡
Wait — KL appears INSIDE the VAE too!
In AR models: we use KL to motivate the training objective, then simplify it away into MLE.
In VAEs: KL appears explicitly as a regularization term in the ELBO — you compute it and backpropagate through it every training step. Different roles!

🔄 AR Training

1

Forward pass

Feed x into model, compute all conditionals p(xᵢ|x<ᵢ) in one pass (with masking).

2

Compute loss

NLL = -Σ log p(xᵢ|x<ᵢ). Exact. No approximation.

3

Backprop & update θ

Standard gradient descent on exact NLL. Simple and clean!

✅
Training is stable and exact. One network, one loss.

🧬 VAE Training

1

Encode x → (μ, σ)

Encoder outputs mean and variance of the approximate posterior q(z|x).

2

Sample z (reparameterization trick)

z = μ + σ·ε where ε ~ N(0,I). This makes sampling differentiable!

3

Decode z → x̂

Decoder reconstructs input from latent code.

4

Compute ELBO loss

Reconstruction loss + KL penalty. Backprop through both encoder and decoder.

⚠️
Training requires balancing two loss terms. Can suffer from posterior collapse (KL term dominates → encoder becomes useless).

Here are the practical trade-offs to help you choose:

Use AR when you need:

  • 🎯 Exact likelihood — e.g., comparing models, data compression
  • 📝 Text generation — natural sequential structure
  • 📊 Density estimation — anomaly detection, hypothesis testing
  • 🔍 Interpretability — each conditional is explicit
  • ✅ Stable training — one simple loss function

Use VAE when you need:

  • 🌌 Latent space navigation — e.g., interpolate between faces
  • ⚡ Fast parallel generation (single decoder forward pass)
  • 🎨 Disentangled representations — control independent factors
  • 🔄 Image-to-image tasks — where conditioning on z matters
  • 🗜️ Compression / representation learning

🏆
The big picture:
Both AR models and VAEs are generative models trying to learn Pdata. They differ in HOW:
  • AR models say: "I'll learn the full joint distribution by factorizing it into a chain of conditionals. No shortcuts, no approximations. Pure MLE."
  • VAEs say: "I'll introduce a latent variable z that captures the essence of the data. I can't compute the exact likelihood but I'll optimize a lower bound. I get a useful latent space in return."
Modern approaches often combine both ideas — e.g., VQ-VAE uses discrete latent codes, then trains an AR model (like PixelCNN) in the latent space!
