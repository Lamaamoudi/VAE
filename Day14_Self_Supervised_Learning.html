Motivation

Why do we need self-supervised learning?

"Give a robot a label and you feed it for a moment; teach a robot to label and you feed it for a lifetime." — Pierre Sermanet
The Problem
  • Annotating data is expensive and slow
  • Each new task often needs a brand-new labeled dataset
  • Requires manuals, annotators, GUIs, storage pipelines
  • High-quality labels can be impractical (medicine, legal)
The Opportunity
  • The internet has vast unlabeled data — images, video, text
  • SSL pre-trains models on this data to extract reusable features
  • Works across vision, language, audio, time-series
  • Mimics how animals and babies learn from the world
🌍
The core insight: Instead of asking humans to label data, we design tasks where the data labels itself. Hide part of an image — the model predicts it. The "answer" already exists in the original image; no human annotation needed.

Evolution of the Field

Generative Models (VAEs, GANs)
Learn data distributions; reconstruct or generate data
Predictive / Pretext Tasks
Rotation, jigsaw, colorization, patch prediction — hand-crafted tasks
Contrastive Methods (CPC, MoCo, SimCLR)
Directly optimize for invariance and instance discrimination
Hybrid / Foundation Models
Combining reconstruction + discrimination at massive scale

Introduction

What is SSL and what can you do with it?

Deep Unsupervised Learning

Learns data representations without labels. A subset of Deep Learning → Representation Learning → Machine Learning.

Self-Supervised Learning (SSL)

Creates supervision from the data itself by designing pretext tasks that generate pseudo-labels — no manual annotation needed.

Learning Outcomes

  • Explain the data efficiency imperative behind SSL
  • Identify key cognitive principles that inspire SSL
  • Compare reconstructive, predictive, and contrastive pretext tasks
  • Implement Denoising Autoencoders & Context Encoders
  • Formulate the InfoNCE loss and train CPC models
  • Describe Instance Discrimination
  • Implement the MoCo framework
  • Critically assess rotation, patch, and view prediction tasks

What is SSL — Key Trade-Offs

  • Pretext Task: Predicting masked patches, future representations (CPC), or distinguishing positive/negative pairs (contrastive)
  • Downstream Task: After pre-training, the encoder is fine-tuned for classification, detection, segmentation, etc.
  • Reconstructive (e.g., autoencoders): capture low-level details, may miss semantics
  • Predictive/Contrastive (e.g., CPC, MoCo): focus on high-level semantic information

Cognitive Principles Behind SSL

🧠
Predictive Coding

The brain optimizes to predict future sensory inputs. SSL mimics this by predicting withheld data parts.

⚡
Error Correction

Learning signals arise from reconstructive or contrastive errors — denoising, inpainting, colorization.

👀
Multi-View Perception

Integrating different sensory views enhances representations — the same scene from multiple angles.

Denoising Autoencoders

Vincent et al. (2008, 2010) — ICML

Core Idea

Corrupt the input, train the model to recover the clean version. The model can't just memorize — it must learn the underlying structure of data.

How It Works

  1. Corrupt input: Add Gaussian noise or mask random pixels of \(x\) to produce \(\tilde{x}\)
  2. Encode: Map corrupted input to latent: \(z = f_\theta(\tilde{x})\)
  3. Decode: Reconstruct the original: \(\hat{x} = g_\phi(z)\)
  4. Loss: Minimize reconstruction error between \(\hat{x}\) and clean \(x\)
  5. Update: Backpropagate — the model learns noise-invariant features
Reconstruction Loss (MSE)
$$\mathcal{L}(\theta, \phi) = \mathbb{E}_{x, \tilde{x}} \left[ \| x - g_\phi(f_\theta(\tilde{x})) \|^2 \right]$$

Emphasizing Corrupted Dimensions

Vincent et al. (2010) showed that computing the loss only on corrupted/masked dimensions \(\mathcal{M}\) forces the model to focus capacity on recovering obscured information:

Masked Reconstruction Loss
$$\mathcal{L} = \sum_{i \in \mathcal{M}} \left( x_i - \hat{x}_i \right)^2$$

\(\mathcal{M}\) = set of corrupted/masked indices

Why It Works

Robust Feature Learning

Features become less sensitive to noise and irrelevant variations — the model learns the data manifold.

Better Generalization

Cannot memorize inputs → learns manifold structure of data → better on unseen/noisy data.

💡
Analogy: If someone erased random parts of your study notes, you'd need to understand the material to fill in the gaps — not just memorize. That's exactly what a denoising autoencoder learns to do!

Context Encoders

Pathak et al. (2016) — CVPR 2016

Core Idea

Randomly mask entire regions of an image and train the model to reconstruct the missing content. To do this well, the model must understand what belongs in that region — forcing semantic representation learning.

Training Objective

Reconstruction Loss (L2)
$$\mathcal{L}_{rec} = \| M \odot (x - G(x \cdot (1{-}M))) \|^2$$

\(M\) = binary mask, \(G\) = encoder-decoder, \(\odot\) = elementwise product

Adversarial Loss (GAN variant)
$$\mathcal{L}_{adv} = \max_D \,\mathbb{E}_x\!\left[\log D(x) + \log(1 - D(G(\hat{x})))\right]$$

Discriminator \(D\) encourages realistic inpainted regions

Why Does This Work?

  • The network must understand global and local context to generate plausible content
  • Forces learning of semantic features and spatial relationships
  • Representations transfer well to classification, detection, segmentation

Restore damaged or removed regions

Pre-trained features for other vision tasks

Anomaly Detection

Predicting One View from Another

Richard Zhang (2019)

Core Idea

Train a model to predict or reconstruct one view of data given another. Views can be different modalities (infrared vs. RGB) or different augmentations (color vs. grayscale).

Relative Position of Image Patches

Doersch et al. (2015) · Noroozi & Favaro (2016)

The Task

Take two image patches and ask the model: where is patch B relative to patch A? (8 possible positions: above-left, above, above-right, left, right, below-left, below, below-right).

Step-by-Step

  1. Divide an image into a 3×3 grid of patches
  2. Sample the center patch as the reference
  3. Sample one of the 8 surrounding patches
  4. Feed both patches through the network
  5. Predict: which of the 8 positions does the second patch occupy?

Jigsaw Puzzle Extension (Noroozi & Favaro, ECCV 2016)

Rather than predicting the position of one patch, shuffle all patches and predict the correct ordering — solving a jigsaw puzzle as a pretext task.

Input

Shuffled 3×3 patches from an image

Prediction

Correct permutation index from a set of \(N\) pre-defined permutations

What the Model Learns

🧠
Geometric relationships and spatial context. The model must understand where heads are relative to bodies, how objects are structured, and what visual patterns coexist. This knowledge transfers well to object detection and segmentation.

Rotation Prediction

Gidaris, Singh, Komodakis — ICLR 2018

The Task

Rotate images by one of 4 predefined angles: \(0°, 90°, 180°, 270°\). Train a classifier to predict which rotation was applied. The rotation label is free — no human annotation required!

Implementation Steps

  1. Randomly select rotation \(r \in \{0°, 90°, 180°, 270°\}\)
  2. Apply rotation to image \(x\) → get \(x^r\)
  3. Feed \(x^r\) through encoder \(f_\theta\)
  4. Pass features through a 4-class classification head
  5. Compute cross-entropy loss and update
Cross-Entropy Loss for Rotation Prediction
$$\mathcal{L} = -\frac{1}{N} \sum_{i=1}^{N} \sum_{r \in \mathcal{R}} \mathbf{1}[y_i = r] \log p(r \mid x_i^r;\, \theta)$$

\(\mathcal{R} = \{0°, 90°, 180°, 270°\}\), \(y_i\) = true rotation applied to image \(i\)

Why Does Rotation Work as a Pretext Task?

To predict a 90° rotation, the model needs to recognize canonical orientations of objects — faces, cars, animals all have strong orientation priors. This forces learning of global shape and semantic features.

Benefits

Simple

Easy to implement, no special infrastructure

Efficient

Computationally inexpensive vs. generative methods

Transferable

Features transfer well to classification & detection

Contrastive Predictive Coding (CPC)

van den Oord, Li, Vinyals — arXiv:1807.03748

Core Idea

Learn representations by predicting future information in latent space using autoregressive models. Instead of predicting raw pixels, CPC predicts in the latent space — a much richer task.

Key Components (click to expand)

Encoder
$$z_t = \text{Encoder}(x_t)$$

Typically a convolutional network for images/audio or a transformer for text. Maps the raw input sequence to latent vectors.

Autoregressor
$$c_t = \text{AR}(z_{\leq t})$$

Often an RNN or masked transformer. Aggregates all past latents \(\{z_1,\ldots,z_t\}\) into a single context vector \(c_t\).

Prediction Step k
$$\hat{z}_{t+k} = W_k \, c_t$$

A linear transformation \(W_k\) (one per future step \(k\)) predicts what the future latent will look like, given the current context.

InfoNCE Loss for prediction step k
$$\mathcal{L}_{\text{InfoNCE}} = -\mathbb{E}\left[\log \frac{\exp\!\left(z_{t+k}^\top W_k c_t\right)}{\displaystyle\sum_{j} \exp\!\left(z_j^\top W_k c_t\right)}\right]$$

Denominator sums over one positive (true future) + multiple negative samples. Minimizing this maximizes mutual information between \(c_t\) and \(z_{t+k}\).

Full Pipeline

Input \(x_1\ldots x_t\)
→
Encoder \(z_t\!=\!f(x_t)\)
→
AR \(c_t\!=\!\text{AR}(z_{\leq t})\)
→
Predict \(\hat{z}_{t+k}\!

InfoNCE Loss

Instance Discrimination & Contrastive Learning

The foundation of modern SSL

Pretext Task Evolution
  • Earlier: handcrafted tasks (jigsaw, colorization, rotation)
  • Now: contrastive learning directly optimizes invariance & discrimination
  • Result: better, more transferable features — narrows gap with supervised learning
Contrastive Learning
  • Uses data structure instead of labels
  • Pulls together positive pairs (same image, different augmentations)
  • Pushes apart negative pairs (different images)
  • Uses InfoNCE or similar contrastive loss

Instance Discrimination — Core Idea

Treat every image as its own class. With \(N\) images there are \(N\) classes. The model must learn to tell apart individual instances.

Positive Pairs ✅

Two different augmented views of the same image (crop, flip, color jitter)

Negative Pairs ❌

Augmented views from different images — treated as different classes

InfoNCE Loss for Instance Discrimination
$$\mathcal{L} = -\log \frac{\exp\!\left(\text{sim}(z_i, z_j)/\tau\right)}{\displaystyle\sum_{k=1}^{2N} \mathbf{1}_{[k \neq i]} \exp\!\left(\text{sim}(z_i, z_k)/\tau\right)}$$

\((i,j)\) = positive pair, \(\tau\) = temperature, \(\text{sim}(\cdot,\cdot)\) = cosine similarity

Methods Built on Instance Discrimination

MoCo

Momentum Contrast — large dictionary queue + EMA key encoder

SimCLR

Simple Framework — large batches + projection head

PIRL

Combines pretext tasks with contrastive learning

Momentum Contrast (MoCo)

He et al. — CVPR 2020

The Problem MoCo Solves

Effective contrastive learning needs a large number of negatives. But batch size is limited by GPU memory. MoCo decouples negative set size from batch size using a dictionary queue.

Two Encoders

Query Encoder \(f_q\)

Processes one augmented view. Updated by standard backpropagation.

Key Encoder \(f_k\)

Processes the other augmented view. Updated by Exponential Moving Average (EMA) of \(f_q\) — NOT by backprop!

EMA Momentum Update
$$\theta_k \leftarrow m \cdot \theta_k + (1 - m) \cdot \theta_q$$

\(m = 0.999\) — the key encoder evolves very slowly, keeping queued keys consistent

Dictionary Queue

MoCo maintains a large FIFO queue of encoded keys from recent batches:

  • Provides a large negative set (e.g., 65,536 negatives)
  • Doesn't require storing all negatives in GPU memory at once
  • Keys stay consistent because the key encoder changes slowly (EMA)
MoCo InfoNCE Loss
$$\mathcal{L}_q = -\log \frac{\exp(q \cdot k^+ / \tau)}{\exp(q \cdot k^+ / \tau) + \displaystyle\sum_{i=0}^{N} \exp(q \cdot k_i / \tau)}$$

\(q\) = query, \(k^+\) = positive key (same image, different augmentation), \(\{k_i\}\) = negative keys from queue, \(\tau\) = temperature

Training Steps

  1. Generate two augmentations: one for \(f_q\) (query), one for \(f_k\) (key)
  2. Encode both: \(q = f_q(\text{aug}_1(x))\), \(k^+ = f_k(\text{aug}_2(x))\)
  3. Compute InfoNCE loss using \(q\), \(k^+\), and the dictionary queue of negative keys
  4. Update \(f_q\) via backpropagation; update \(f_k\) via momentum EMA (no gradient!)
  5. Enqueue new key; dequeue oldest key to maintain dictionary size
🎯
Why momentum matters: If the key encoder changed rapidly, keys in the queue would become inconsistent — encoded by very different models. EMA momentum keeps the key encoder evolving slowly so all queued keys are encoded by a similar model.
